why ingress?

what is ingress?
🧠 In Layman Terms
Think of your house 🏠:
• You have many rooms inside (like your app services)
• But people from outside can’t directly walk into any room
• They enter through the main gate or door
👉 That main door = Ingress
So:
• Outside users (internet) → come to one entry point
• That entry point decides which room (service) they should go to
Example:
If someone visits:
• myapp.com/login → go to login service
• myapp.com/products → go to product service
👉 One door, smart routing inside

1. Ingress is incoming traffic to ur application.
2. Ingress is an API object that manages external HTTP/HTTPS access to services inside a cluster.
🔹 Problem it solves
Without Ingress:
• Each service needs its own LoadBalancer (costly + messy)
With Ingress:
• One entry point
• Routes traffic to multiple services

NOTE:- User → Load Balancer → Ingress Controller → Service → Pods

Step 1: 🌐 User → Load Balancer
• Cloud LB receives request
Step 2: 🚦 Load Balancer → Ingress Controller
• Forwards traffic into cluster
Step 3: 🧠 Ingress Controller checks rules
• Reads Ingress Resource
• Sees:
• /api → api-service
Step 4: 🔀 Routes to Service
• Sends request to correct service
Step 5: ⚙️ Service → Pod
• Load balances across pods
Step 6: ✅ Response goes back same path

Ingress is a Kubernetes resource that manages external access to services inside the cluster.
It lets you route traffic (like HTTP/HTTPS) to different services based on URL or domain.
Instead of exposing many services, you use one entry point (Ingress) to control everything.
 
NOTE:- Ingress resource is useless, unless and until we have ingress controller deployed in our k8s cluster.

Step 1:- Deploy the ingress controller(basically developed by load balancers companies) using helm, which ingress-controller to use will be usually decided by the devops, devs and architects and other teams. It is a single time installation to your cluster.
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/cloud/deploy.yaml
NOTE:- Ingress controller is exposed using aservice type loadblancer that triggers cloud provider load blancer
A Loadbalancer service type is automatically created when we deploy ingress controller, when k8s sees this loadbalancer service it asks the Cloud provider to create load balancer.

Step 2:- Create a deployment and deploy it.
Step 3:- create a clusterIP service and deploy it
Step 4:- create a ingress service and deploy it
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
 name: my-app-ingress
spec:
 ingressClassName: nginx
 rules:
 - host: myapp.com
   http:
     paths:
     - path: /
       pathType: Prefix
       backend:
         service:
           name: my-app-service
           port:
             number: 80
NOTE:- ingressClassName is mandatory because we are telling the cluster to use the nginx ingress controller for routing when there are multiple imgress controllers in the namespaces.

K8S TROUBLESHOOTING:-

IMGPULLBACKOFF:-
1. Two cases where you will face this error:-
 1.1 Invalid image name, non existent image
 1.2:- trying to pull the images from private registry
2. The first error message you see is not "imgpullbackoff", it is "errorimgpull". what k8s does is when it notices errorimagepull, it will wait for sometime like 10sec and try to pull the image because sometimes there might be network issues when pulling the image, so it will try again in a incremental way, next time it will wait for 20sec and try again and it keeps trying. Now the "errimgpull" is converted to "imgpullbackoff"
3. To resolve error 1.1 make sure you provide correct image by checking the commands such as describe, events.
4. To resolve error1.2 you must create "imagepullsecret" that contains the private registry credentials.
5. In eks, to resolve 1.2 If the image is being pulled by a kubelet that is on a node that is in private subnet, then "impullsecrets" will not be required, it will need 2 permissions:-
 5.1 the node IAM role must have roles/permission to pull the image from ECR
 5.2 the node must have internet access to pull the image but as it is in VPC private subnet we can create VPC endpoints.

NOTE:-
Why IRSA cannot be used for image pulls
The key reason is Container images are pulled before the Pod starts, but IRSA only works after the Pod is running.

CRASHLOOPBACKOFF:-

1. A pod gets scheduled onto a node by the scheduler, kubelete in the node will pull the image and update the status of the pod from "pending" to "container creating".
2. From the "container creating" status the pod can move to either "running" or "error" status.
3. From "running/error" status it can move to "crashed" state and k8s will try to restart the pod in an incremental limit, it will again go to "container creation" and this is a loop

Main reasons for crashloopbackoff error:-
1. providing wrong command line arguments eg:- passing app1.py instead of app.py in "CMD python3 app.py"
